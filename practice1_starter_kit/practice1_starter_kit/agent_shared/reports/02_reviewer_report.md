## 2. 과대 범위

- **이력 목록(history list) 렌더링**: "새로고침 후에도 이전 입력 목록을 다시 볼 수 있어야 한다"는 답변은 단순 "마지막 상태 복원"이 아니라 **여러 건을 리스트로 표시**하는 것으로 읽힌다. Builder 계획(04)이 원래 가정한 "단일 결과 카드 렌더링"보다 UI/로직이 한 단계 더 크다(목록 아이템 렌더링, 정렬, 개수 제한 등 필요).
- **저장 계층 설계**: `localStorage` 저장/조회, 손상 데이터 방어, 스키마 관리는 Research 보고서가 "MVP 최소 범위"로 제시한 "판정 함수 + 렌더링"보다 명백히 넓은 작업이다.
- **Builder 04 문서의 classifier.js 자동 분류 설계 잔존**: 04 문서 3~4번 섹션은 여전히 `determinePriority(text)`, `determineRole(text)`, `matchedKeywords` 등 키워드 기반 자동 판정을 전제로 한 데이터 구조/함수를 담고 있다. 이는 이미 도착한 교육생 답변(수동 선택)과 모순되므로, 그대로 구현하면 사용하지 않을 코드를 만드는 과대 범위가 된다.
- 위 세 가지를 합치면 "폼 입력 + 저장 + 목록 렌더 + 방어 로직"까지 다뤄야 하므로, 애초 Research가 추천한 "1~2시간 규칙 기반 판정기"보다 실제 작업량이 늘어난 상태다.

## 3. 애매한 요구사항

1. **이력 목록의 형태 미정**: 몇 건까지 보여줄지, 정렬 순서(최신순/등록순)가 무엇인지, 개별 항목 삭제/전체 초기화 기능이 있는지 답변에 없음.
2. **"고정 목록"의 실제 값 미정**: 담당 역할/다음 액션의 정확한 문자열 목록이 아직 확정되지 않았다(Builder 04는 "Frontend, Backend, QA, Infra 등" 예시만 제시).
3. **필수 입력 범위 불명확**: "빈 입력은 저장하지 않는다"는 답변이 텍스트 필드만을 말하는지, 우선순위/역할/액션 미선택 시에도 저장을 막는지 불명확.
4. **UI 컨트롤 타입 미정**: 우선순위 선택이 라디오 버튼인지 드롭다운인지 확정되지 않음(답변은 "예: 라디오/드롭다운"으로 둘 다 열어둠).
5. **localStorage와 테스트 환경의 관계 미정**: Tester가 Node.js 스크립트로 테스트한다고 했을 때, `window.localStorage`가 없는 순수 Node 환경에서 어떻게 테스트할지에 대한 설계가 어느 보고서에도 없다.
6. **실행 환경(file:// vs 정적 서버)**: Research 보고서가 이미 "단정하지 않음"으로 남긴 사항인데, localStorage는 `file://`로 직접 여는 브라우저 환경에서 동작이 제한되거나 브라우저별로 다르게 동작할 수 있어 이번 기능 추가로 이 미정 사항의 중요도가 더 커졌다.

- **P0-2. Research 제외 권고 항목의 재포함**: localStorage 이력 저장은 01_research_report.md 4번 섹션에서 명시적으로 실습 범위 제외 대상으로 분류한 "이력 저장(DB/파일 영속화)"에 해당한다. 시간 제약이 있는 실습에서 제외됐던 항목이 되돌아온 것이므로, Main Lead가 시간 재배분(다른 기능 축소 등) 여부를 결정해야 한다.
- **P0-3. 테스트 아키텍처 미해결**: localStorage(브라우저 전용 API)를 Node.js 기본 모듈만으로 작성되는 테스트 스크립트가 다루려면, 저장 로직을 "주입 가능한 storage 인터페이스"로 분리하거나 손수 작성한 최소 mock이 필요하다. 이 설계가 Builder/Tester 사이에 합의되지 않은 채 각자 진행하면, 테스트 단계에서 구조 재작업이 발생할 가능성이 크다.

## 5. Builder 반영 조건

1. `classifier.js`의 텍스트 키워드 기반 자동 판정 함수(`determinePriority(text)`, `determineRole(text)` 등)와 `matchedKeywords` 필드는 폐기하거나, "사용자가 고정 목록에서 고른 값을 그대로 전달"하는 단순 매핑으로 대체할 것. 교육생 답변(수동 선택)과 어긋나는 자동 분류 코드는 만들지 않는다.
2. localStorage 접근 로직은 반드시 `getItem/setItem` 형태의 최소 인터페이스로 감싸서, 실제 `window.localStorage`와 테스트용 mock 객체를 서로 바꿔 끼울 수 있는 구조로 설계할 것.
3. 이력 목록은 개수 상한(예: 최근 N건)과 정렬 기준(예: 최신순)을 명시적으로 정의하고 문서화할 것. 삭제/초기화 기능은 이번 범위에 포함하지 않는다고 계획서에 명시할 것(범위 확대 방지).
4. 우선순위 선택 UI 타입(라디오 또는 드롭다운), 역할/다음 액션 고정 목록의 정확한 문자열 목록을 구현 착수 전에 확정해 계획서에 명문화할 것.
5. "빈 입력이라 저장하지 않는" 조건에서 어떤 필드가 필수인지(텍스트만인지, 선택 항목 포함인지)를 명시할 것.
6. localStorage에서 값을 불러올 때 JSON 파싱 실패/형식 불일치에 대비한 방어 코드(try/catch + 기본값 폴백)를 계획에 포함할 것.

## 6. Tester 반영 조건

1. localStorage에 의존하는 로직을 테스트하려면, Node.js 기본 모듈만으로 작성한 최소 in-memory mock(예: `{ getItem, setItem, removeItem }`을 흉내 내는 순수 객체)을 직접 만들어 사용할 것 — jsdom 등 외부 패키지 사용 금지 원칙을 지킬 것.
2. 테스트 대상을 "순수 로직 함수(이력 추가/정렬/개수 제한 등)"와 "DOM 렌더링"으로 분리하고, 자동화 테스트는 전자에 집중, 후자는 수동 확인 항목으로 분리할 것을 권장.
3. 엣지 케이스에 빈 입력, 손상된 localStorage 데이터, 이력 개수 상한 초과, 고정 목록에 없는 값 복원 시나리오를 포함할 것.
4. 저장/조회 테스트가 실제 브라우저 없이 실행 가능한 구조인지(즉 Builder 반영 조건 2번이 지켜졌는지)를 테스트 계획 확정 전에 Builder와 반드시 교차 확인할 것.
