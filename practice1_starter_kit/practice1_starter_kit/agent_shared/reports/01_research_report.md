# 01 Research Report

## 1. 아이디어 재정의
코드 변경 요청(텍스트)을 입력하면, 정해진 규칙에 따라 우선순위(P0/P1/P2), 리뷰 필요 여부, 담당 역할, 다음 액션을 한 화면에 즉시 보여주는 1페이지 웹앱을 만든다.

## 2. 가능한 구현 방식

### 방식 A: 정적 HTML + 인라인 JS (규칙 기반 분류, 서버 없음)
- 입력 textarea 하나 + 결과 표시 영역 하나로 구성된 단일 HTML 파일.
- 브라우저 내 JavaScript에서 키워드/패턴 매칭(정규식, 문자열 포함 여부)으로 P0/P1/P2, 리뷰 필요 여부, 담당 역할, 다음 액션을 계산해 즉시 렌더링.
- 예: "장애", "긴급", "hotfix", "보안" 등의 키워드가 있으면 P0로 분류하는 식의 규칙 테이블.
- 서버, 빌드 도구, 외부 패키지 전혀 불필요. 브라우저에서 파일을 열기만 해도 동작.

### 방식 B: Node.js 내장 http 모듈 기반 소형 서버 + 정적 페이지
- Node.js `http` 모듈만으로 최소 서버를 띄우고, 클라이언트가 POST로 텍스트를 보내면 서버 쪽에서 동일한 규칙 기반 분류 로직을 수행해 JSON으로 응답.
- 프론트엔드는 fetch API로 결과를 받아 1페이지에 표시.
- 서버 로직과 클라이언트 로직을 분리하고 싶을 때 적합하지만, 방식 A 대비 구현/실행 단계가 하나 더 늘어난다(서버 기동, 포트 관리).

### 방식 C: 규칙 테이블을 별도 JSON/JS 설정 파일로 분리 + 방식 A 로직 재사용
- 방식 A와 동일한 정적 HTML 구조를 쓰되, 분류 규칙(키워드→우선순위/역할/액션 매핑)을 별도의 설정 객체(JS 배열/객체)로 분리.
- 규칙을 코드 로직과 분리해두면 실습 중 규칙을 조정하거나 확장하기 쉬워지고, 실습 2에서 Builder가 구현할 때 "규칙 데이터"와 "판정 함수"의 책임을 나누기 쉬워진다.
- 여전히 외부 패키지, 서버 불필요.

## 3. 추천 MVP
**방식 A 또는 방식 C(사실상 방식 A의 세부 구조 개선안)를 추천한다.**

최소 MVP 범위:
- 입력: textarea 1개(코드 변경 요청 텍스트를 자유 입력).
- 버튼 1개(예: "분석하기") 또는 입력 시 자동 실행.
- 출력 영역 1개에 아래 4개 항목을 표시:
  1. 우선순위: P0 / P1 / P2 (규칙 기반 키워드 매칭 결과, 매칭 없으면 기본값 P2 또는 "미분류")
  2. 리뷰 필요 여부: 예/아니오 (예: P0/P1이면 "예", P2면 "아니오" 같은 단순 규칙과 연동 가능)
  3. 담당 역할: 규칙에 매핑된 역할 이름(예: "백엔드", "프론트엔드", "인프라" 등 텍스트 값, 실제 조직 역할명 아님)
  4. 다음 액션: 규칙에 매핑된 짧은 텍스트 안내(예: "즉시 리뷰 요청", "다음 스프린트 반영 검토")
- 판정 로직: 사전 정의된 키워드 목록과 우선순위/역할/액션 매핑 테이블을 순서대로 검사하는 단순 규칙 기반 로직(if/switch 또는 배열 순회 + 정규식).
- 서버, DB, 외부 API, 외부 npm 패키지 없음. 단일 HTML(+ 내장 script) 파일로 완결.

이 범위는 1~2시간 실습 내에서 "규칙 테이블 정의 + 판정 함수 작성 + 결과 렌더링"만으로 완결 가능하다고 판단된다.

## 4. 제외할 기능
- 실제 코드 diff 파싱, 정적 분석, AST 기반 분석 (범위 과다, 실습 시간 내 불가능에 가까움)
- 실제 LLM/외부 API 호출을 통한 분류(이번 실습은 "규칙 기반" 범위로 한정, 외부 연동은 실습 시간과 네트워크/키 의존성 리스크를 늘림)
- 사용자 인증, 다중 사용자, 이력 저장(DB/파일 영속화)
- 담당자 자동 배정을 위한 실제 조직도/실제 팀원 매핑 연동
- 알림(이메일/슬랙 등) 발송 기능
- 다국어 지원, 반응형 디자인 고도화 등 UI 부가 기능
- 규칙 테이블의 자동 학습/업데이트(고정된 정적 규칙만 사용)

## 5. 실습 난이도 판단
- 방식 A(정적 HTML + 인라인 JS, 규칙 기반) 기준으로 1~2시간 실습 범위 내에서 구현 가능하다고 판단된다.
- 근거: 서버/빌드/외부 패키지가 필요 없고, 핵심 로직이 "키워드 매칭 후 매핑 테이블 조회"라는 단순한 형태이며, UI도 입력창 1개 + 출력 영역 1개로 최소화할 수 있음.
- 다만 규칙 테이블(키워드 → 우선순위/역할/액션 매핑)의 설계와 케이스 수를 얼마나 늘리느냐에 따라 소요 시간이 달라질 수 있어, 초기에는 규칙 항목 수를 5~10개 수준으로 제한하는 것이 안전해 보인다.
- 방식 B(Node http 서버 분리)는 기능적으로는 유사하지만 서버 기동/포트 관리 등 부가 절차가 늘어나므로, 시간이 부족할 경우 방식 A보다 리스크가 크다고 판단된다.

## 6. Assumptions
- "우선순위/리뷰 필요 여부/담당 역할/다음 액션" 판정은 실제 조직의 프로세스나 실제 데이터가 아니라, 실습용으로 임의 설계한 규칙 테이블(가상의 키워드-분류 매핑)을 사용한다고 가정한다.
- "담당 역할"은 실제 팀/조직 구조가 아니라 예시성 텍스트 값(예: 프론트엔드/백엔드/인프라 등)으로 가정한다.
- 사용자가 입력하는 "코드 변경 요청" 텍스트의 형식(자유 텍스트 vs 구조화된 필드)은 자유 텍스트로 가정했으며, 실제 채택 여부는 Reviewer/Tester 검토에서 확정될 수 있다.
- 규칙 매칭에서 여러 키워드가 동시에 매칭될 경우의 우선순위 결정 방식(예: "가장 높은 우선순위를 채택")은 본 리서치에서 확정하지 않았으며, Builder 설계 단계에서 구체화가 필요하다고 가정한다.
- 브라우저 실행 환경(로컬 파일 직접 열기 vs 간단한 정적 서버로 서빙)에 대한 제약은 실습 환경에 따라 달라질 수 있어 단정하지 않았다.
