## 1. 테스트 대상 요약

`03A_test_answers.md`에서 확정된 답변에 따르면, 우선순위(P0/P1/P2)와 리뷰 필요 여부는
**자동 판정 로직이 아니라 사용자가 직접 선택/체크하는 값**이고, 담당 역할과 다음 액션은
**고정 목록에서 선택**하는 값이다. 따라서 이 앱은 "규칙 기반 분류 엔진"이 아니라
**사용자 입력을 검증하고, 그대로 저장하고, 목록으로 표시하는 폼 + 리스트 앱**이며,
테스트의 무게중심도 "분류 로직 정확성"이 아니라 아래 3가지로 옮겨간다.

1. **빈 입력 검증 로직** — 빈 문자열/공백만 있는 입력은 저장하지 않고 안내만 표시한다 (질문 1 답변).
2. **입력값을 그대로 저장/표시하는 로직** — 사용자가 고른 우선순위(P0/P1/P2, 질문 2),
   체크박스로 입력한 리뷰 필요 여부(질문 3), 고정 목록에서 고른 담당 역할/다음 액션(질문 4)을
   가공/재판정 없이 레코드 객체에 그대로 담아 화면에 표시하는 로직.
3. **localStorage 저장/조회 로직** — 새로고침 후에도 이전 입력 목록(우선순위/리뷰 필요 여부/
   담당 역할/다음 액션 포함)을 순서대로 다시 볼 수 있어야 한다 (질문 5 답변).

키워드 기반 자동 우선순위 분류, 리뷰 필요 자동 판정 로직은 **테스트 대상에서 제외**한다
(교육생 답변에 따라 해당 로직 자체가 구현되지 않으므로).

이 3가지는 "입력 객체 → 검증 → 정규화된 레코드 객체 → 저장/조회" 형태의 순수 함수로
분리 가능하다고 가정하며, Node.js 기본 `assert`(또는 `node:test`) 모듈만으로 UI 없이
로직 단위 테스트가 가능하다. localStorage는 브라우저 전용 API이므로, 브라우저 없이 테스트하려면
`{getItem, setItem}`을 흉내내는 간단한 in-memory mock 객체(순수 JS 객체, 외부 패키지 불필요)를
저장소 함수에 주입하는 방식을 제안한다.

## 2. 테스트 질문 5개

1. 빈 입력(빈 문자열 또는 공백만 있는 입력)은 저장/분석하지 않고 에러 또는 안내만 표시하는 것이 맞습니까? 아니면 기본값(P2 등)으로 처리합니까?
2. 우선순위 P0/P1/P2 분류는 키워드 기반 규칙(예: "장애", "긴급", "hotfix" 포함 시 P0)으로 자동 분류합니까, 아니면 사용자가 직접 선택하는 수동 입력입니까? 자동이라면 규칙(키워드 목록)을 지금 정해줄 수 있습니까?
3. 리뷰 필요 여부는 어떤 기준으로 판단합니까? (예: 특정 우선순위 이상이면 무조건 리뷰 필요, 또는 특정 키워드가 있으면 필요, 또는 항상 필요)
4. 담당 역할 추천과 다음 액션은 고정된 목록 중에서 고르는 것입니까(예: 역할 = [Frontend, Backend, QA, Infra] 중 하나, 액션 = [즉시 수정, 리뷰 요청, 백로그 등록] 중 하나), 아니면 자유 텍스트로 생성합니까?
5. 결과를 localStorage 등에 저장해서 새로고침 후에도 이전 입력 목록을 다시 볼 수 있어야 합니까, 아니면 이번 실습 2 MVP는 "입력 → 즉시 결과 표시"까지만 이고 저장/기록 기능은 범위 밖입니까?

## 3. 예상 JS 테스트 파일

실습 2 구현이 로직을 별도 모듈(예: `src/record.js`, `src/storage.js`)로 분리한다고
가정할 때의 예상 테스트 파일 구조 (아직 작성하지 않음, 계획만):

```
test/
  validate-input.test.js  // 빈 문자열/공백 입력 검증 로직 (질문 1)
                          //   - 빈 입력 → 저장 거부 + 안내 플래그 반환
                          //   - 공백만 있는 입력 → 저장 거부 + 안내 플래그 반환
                          //   - 유효한 텍스트 → 통과

  build-record.test.js    // 사용자 선택값을 그대로 담는 레코드 생성 로직 (질문 2·3·4)
                          //   - priority(P0/P1/P2 중 사용자가 고른 값)를 가공 없이 그대로 보존
                          //   - reviewNeeded(체크박스 true/false)를 그대로 보존
                          //   - role/nextAction이 고정 목록(허용된 값 집합) 안의 값인지 검증
                          //   - 고정 목록에 없는 값이 들어오면 에러 또는 거부 (방어 코드 테스트)

  storage.test.js         // localStorage 저장/조회 로직 (질문 5)
                          //   - in-memory mock storage에 레코드 저장 후 다시 읽으면 동일한 배열이 반환됨
                          //   - 여러 건 저장 시 입력 순서(또는 정해진 정렬 기준)대로 조회됨
                          //   - 새 레코드 추가 시 기존 레코드가 삭제되지 않고 누적됨
                          //   - storage가 비어있을 때 조회하면 빈 배열을 반환함 (에러 아님)
```

- 실행 방식: 외부 프레임워크 없이 Node.js 내장 `assert`(또는 `node:test`) 모듈만 사용.
  예: `node --test test/` 또는 `node test/validate-input.test.js` 형태로 실행.
- UI(DOM) 테스트는 실습 시간상 범위 밖으로 두고, 순수 로직 함수(검증/레코드 생성/저장-조회)만
  테스트 대상으로 삼는다.
- (제거됨) 초안에 있던 "키워드 기반 자동 우선순위/리뷰 필요 여부 분류" 테스트는 답변에 따라
  해당 기능이 구현되지 않으므로 계획에서 제외한다. `classify.test.js`, `format-output.test.js`는
  `build-record.test.js`로 대체한다.

## 5. PASS/FAIL 기준

- PASS: 각 테스트 파일의 모든 `assert` 호출이 예외 없이 통과하고, 프로세스 종료 코드가 0이다.
- FAIL: `assert` 실패(AssertionError) 발생, 또는 함수가 예상 타입/값과 다른 결과를 반환.

구체적 기준:

1. **빈 입력 검증 (질문 1 반영)**
   - PASS: 빈 문자열 또는 공백만 있는 입력이 주어지면 검증 함수가 "저장 불가"를 나타내는 값
     (예: `false` 또는 에러 객체)을 반환하고, 저장 함수가 호출되지 않는다.
   - PASS: 공백이 아닌 유효한 텍스트는 검증을 통과한다.
   - FAIL: 빈/공백 입력인데 저장이 진행되거나, 유효한 입력인데 거부되는 경우.

2. **레코드 생성 (질문 2·3·4 반영 — 자동 분류 없음)**
   - PASS: 레코드 생성 함수는 사용자가 선택한 `priority`, `reviewNeeded`, `role`, `nextAction`
     값을 재해석/재판정 없이 입력받은 그대로 레코드에 담아 반환한다.
   - PASS: `priority`는 반드시 `"P0"|"P1"|"P2"` 중 하나여야 하고, 그 외 값이 들어오면
     에러를 던지거나 저장을 거부한다 (고정 값 검증).
   - PASS: `role`, `nextAction`은 사전에 정의한 고정 목록에 포함된 값이어야 하며, 목록에
     없는 값은 에러 또는 거부 처리한다.
   - PASS: `reviewNeeded`는 boolean 타입이어야 한다.
   - FAIL: 함수가 입력값을 임의로 변경(예: 다른 priority로 덮어씀)하거나, 고정 목록 밖의
     값을 그대로 통과시키는 경우.

3. **localStorage 저장/조회 (질문 5 반영)**
   - PASS: 저장이 비어있는 상태에서 조회하면 빈 배열을 반환한다 (에러 아님).
   - PASS: 레코드를 저장한 뒤 조회하면 저장한 필드값과 정확히 일치하는 레코드가 반환된다.
   - PASS: 여러 건을 연속 저장해도 이전 레코드가 유실되지 않고 누적된다.
   - PASS: 저장 함수 호출 후 별도의 새 조회 호출(새로고침 시뮬레이션)에서도 동일한 데이터가
     유지된다.
   - FAIL: 조회 시 레코드 누락/순서 뒤바뀜(정렬 기준 미정의 시 보류)/타입 손상(문자열이
     객체로 깨지는 등)이 발생하는 경우.
